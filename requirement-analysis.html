Requirements

Requirements identification is the first step of any software development project. Until the requirements of a client have been clearly identified, and verified, no other task (design, coding, testing) could begin.

1. Functional requirements (FRs)

These describe the functionality of a system — how a system should react to a particular set of inputs and what should be the corresponding output.

Examples:

  • Authentication: "The system must allow users to log in using their email and password".
  • Data Processing: "The system shall calculate the total cost including tax upon checkout".
  • Workflow: "The system must send a confirmation email after an order is placed".
  • User Action: "The user shall be able to search for products by name, category, or price".

2. Non-functional requirements (NFRs)

They are not directly related to what functionalities are expected from the system.

  • Performance: How fast does the system return results?
  • Scalability: How much will performance change with higher workloads?
  • Portability: Which hardware, operating systems, and browsers does the software run on?
  • Compatibility: Does the system conflict with other applications and processes?
  • Reliability: How often does the system experience critical failures?
  • Maintainability: How much time does it take to fix the issue when it arises?
  • Availability: How long is the average system downtime?
  • Security: How well are the system and its data protected against attacks?
  • Usability: How easy is it to use the system?

Practical No.4

Aim: To create SRS (Software Requirement Specification) for a given case study.

Objective:

  1. To understand the structure and purpose of a Software Requirement Specification (SRS) document.
  2. To identify and document functional and non-functional requirements of a software system.
  3. To learn how to clearly define system scope, constraints, assumptions, and user requirements.
  4. To prepare a well-structured SRS document based on IEEE standards.

Theory:

A Software Requirements Specification (SRS) is a comprehensive document that outlines the functional and non-functional requirements of a software system. It serves as a contract between stakeholders, ensuring clarity and alignment on the software's objectives, features, and constraints.
